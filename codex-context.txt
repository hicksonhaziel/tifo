TIFO Codex Context
Last updated: 2026-07-02

Purpose
TIFO is a Pear Electron desktop app for a peer-to-peer football "living terrace".
The demo goal is not normal chat. The final demo should show several fans joining
the same match room, creating chat/reaction/chant events, merging them into an
Echo timeline, and replaying a goal moment later.

What Is Implemented Now
- Repo is based on Holepunch's hello-pear-electron starter.
- App is renamed/branded as TIFO.
- Native app identity and Linux dock icon are set up.
- TIFO logo/icon/banner assets live under assets/brand/.
- Electron menu bar is removed.
- Renderer UI has a Home / Join Room screen.
- Renderer UI has a Match Room screen.
- Local-only chat, reactions, and timeline placeholders work inside the renderer.
- Renderer-to-worker JSON command protocol is implemented.
- The worker starts and sends app:ready.
- Fake local room join/leave now happens through workers/main.js.
- Chat, reaction, and Echo replay preview events round-trip through the worker.
- Unknown worker commands return structured error messages.
- Hyperswarm room discovery is implemented.
- Phase 3 originally used framed peer-to-peer JSON event exchange.
- Room code is converted to a deterministic 32-byte Hyperswarm topic.
- Phase 5 room sockets now use Protomux so a control channel and Autobase feed
  replication can share the same Hyperswarm connection.
- Peers exchange room identity messages containing Autobase base and writer keys.
- If two peers created different Autobase bases for the same room, the lower
  base key wins and the other peer reopens on that shared base.
- Existing local events are queued and re-appended after a peer adopts the shared
  base and becomes writable.
- Writable peers append writer:add records so Autobase can call host.addWriter()
  for new peer writer feeds.
- Local room events are appended to the local Autobase writer feed.
- Remote room events are now expected to arrive through Autobase feed replication
  and materialized view updates, not through JSON event broadcasts.
- Remote events are sanitized, deduplicated by id, and displayed in the timeline.
- Leaving a room closes the active room sockets because Hyperswarm leave() does
  not close existing connections by itself.
- Local room events were previously persisted to a per-room Hypercore through
  Corestore in Phase 4.
- Phase 5 uses an Autobase-backed Echo timeline for accepted room events.
- Room history is loaded from the Autobase materialized view when rejoining with
  the same storage path.
- New local and accepted remote events are appended to the Echo timeline.
- Phase 5 close/reopen null-key guard is implemented in TifoEchoTimeline.
- Room UI has been cleaned up into a clearer match dashboard with stronger sync
  state, match summary, chat, action, and Echo timeline sections.
- The room UI shows live P2P state, saved event count, peer-event count, and
  local/peer/saved event badges.
- BUILD.md and PLAN.md are private local planning files and ignored by git.
- package.json uses CommonJS and vanilla JS/CSS. No Tailwind and no TypeScript yet.

What Is Not Implemented Yet
- Persisting the adopted shared Autobase base key for future restarts is not
  implemented yet; a peer can adopt again when it reconnects to another peer.
- No audio chant recording yet.
- No replay Echo mode yet.
- No offline/reconnect demo yet.
- No production release/publishing flow yet.

Current Architecture
- renderer/ is the UI. It should only talk through window.bridge.
- electron/main.js is the Electron main process. It launches Bare workers through
  PearRuntime.run() and forwards worker IPC to the renderer.
- workers/main.js is the Bare worker. This is where P2P, storage, Hyperswarm,
  Corestore, Hypercore, and Autobase code should live.
- The renderer should not know Hyperswarm details.
- The worker should not know DOM details.
- The main process should stay mostly a bridge, not become the app backend.

Pear Docs Checked
- Pear docs say the hello-pear-electron template is split into renderer/,
  workers/main.js, electron/main.js, electron/preload.js, package.json, and build/.
- Pear docs say the renderer calls window.bridge, not ipcRenderer directly.
- Pear docs say worker IPC messages are raw bytes and it is valid to send JSON
  strings over writeWorkerIPC/onWorkerIPC using a message type field.
- Pear docs say P2P app logic belongs in the Bare worker, where Hyperswarm and
  Corestore are available.
- Pear docs describe a production chat shape with a separate Bare worker and
  Autobase-backed shared room.
- Autobase docs say writer nodes are linearized into an eventually consistent
  view and host.addWriter() adds writer keys to the base.
- Corestore/Hypercore docs say replication can attach to a Protomux protocol
  stream, which allows replication and TIFO control messages on one socket.
- Reference URLs:
  https://docs.pears.com/getting-started/from-a-template/start-from-hello-pear-electron/
  https://docs.pears.com/getting-started/build-a-peer-to-peer-chat/reshape-into-a-production-app/
  https://docs.pears.com/reference/building-blocks/autobase/
  https://docs.pears.com/

How To Run
- Single instance:
  npm start
- Multiple demo identities should use separate storage:
  npm start -- --storage /tmp/tifo-amina
  npm start -- --storage /tmp/tifo-yassine
  npm start -- --storage /tmp/tifo-salma
- Reset demo storage only when intentionally starting fresh:
  rm -rf /tmp/tifo-amina /tmp/tifo-yassine /tmp/tifo-salma

Phase 2 Status
Goal completed: the UI talks to the Bare worker through a small JSON protocol
before real networking is added.

Renderer sends these command messages:
- profile:set
- room:join
- room:leave
- chat:send
- reaction:send
- echo:replay

Worker sends these event messages:
- app:ready
- room:joined
- room:left
- peer:count
- sync:status
- event:added
- event:list
- error

Phase 2 implementation notes:
- Keep the existing updater strings working: pear:applyUpdate, pear:updateApplied,
  updating, updated.
- For TIFO messages, send JSON.stringify({ type, ...payload }) over the existing
  worker IPC bridge.
- In the renderer, parse JSON worker messages first. If parsing fails, pass the
  string to existing updater handling.
- In the worker, parse JSON commands. If parsing fails and it is not an updater
  message, log or return a structured error.
- Validate required fields at the worker boundary.
- Maintain room/session state in workers/tifo-room-state.js for Phase 2, even if
  it is fake.
- Renderer state should be updated from worker events, not from direct local
  assumptions.

Phase 2 Exit Gate
- Worker startup emits app:ready. Done.
- Joining a fake local room goes renderer -> worker -> renderer. Done.
- Leaving the room goes renderer -> worker -> renderer and clears room state. Done.
- Chat sends renderer -> worker -> renderer and appears in the timeline. Done.
- Reaction sends renderer -> worker -> renderer and appears in the timeline. Done.
- Unknown commands return a structured error. Done.
- Worker exit/restart does not leave the UI pretending it is connected. Done.

Important Product Rules
- No backend server.
- No accounts.
- No cloud database.
- No sports API dependency for the core demo.
- Desktop-first.
- Keep network behavior visible in the UI: peer count, sync state, local/pending
  state, reconnect/catch-up state.
- Do not add Tailwind or TypeScript until there is a strong reason. The current
  starter is vanilla JS/CSS and this keeps Phase 2 simpler.
